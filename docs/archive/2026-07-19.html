🧠 Data Science Topic of the Day

🎓 Random Forests

Wisdom of crowds, applied to decision trees — robust, interpretable, and remarkably hard to break.

What It Is

A Random Forest is an ensemble learning method that builds a large number of decision trees during training and merges their outputs to produce a more accurate and stable prediction than any single tree could. Each tree is trained on a random bootstrap sample of the data (bagging), and at each split, only a random subset of features is considered. The final prediction is the majority vote (classification) or average (regression) across all trees. The 'randomness' is the secret weapon: it decorrelates the trees so their errors don't compound, dramatically reducing variance without inflating bias. Random Forests are among the most battle-tested algorithms in practical ML — they handle missing values gracefully, resist overfitting, require almost no preprocessing, and produce a native feature importance ranking that is genuinely useful for business storytelling.

How It Works

1. BOOTSTRAP SAMPLING: Draw N random samples with replacement from the training set to create T separate datasets, one per tree. Each dataset contains ~63% of unique records; the rest ('out-of-bag' samples) become a free validation set. 2. RANDOM FEATURE SUBSETS: At every node split within a tree, consider only a random subset of √p features (classification) or p/3 features (regression), where p is total features. This is the key difference from plain bagged trees — feature randomness breaks the dominance of strong predictors and forces diversity across trees. 3. GROW DEEP, DON'T PRUNE: Individual trees are grown to maximum depth (low bias, high variance). The ensemble averages out the variance. 4. AGGREGATE PREDICTIONS: For classification, take the majority vote across all T trees. For regression, take the mean predicted value. 5. OUT-OF-BAG ERROR: Each tree sees ~37% of records it was never trained on. Averaging OOB error across trees gives an unbiased generalization estimate — no separate validation split required. 6. FEATURE IMPORTANCE: Measure the average decrease in node impurity (Gini or entropy) caused by each feature across all trees. Features that split high up and frequently get high importance scores. Permutation importance (shuffling a feature and measuring accuracy drop on OOB data) is a more reliable alternative. Key hyperparameters: n_estimators (more is better until diminishing returns, ~200-500), max_features, max_depth, min_samples_leaf. Scikit-learn's defaults are surprisingly good as a starting point.

When To Use

USE IT WHEN: (a) You need a strong baseline fast — Random Forests beat linear models on tabular data most of the time with zero feature engineering. (b) Your dataset has a mix of numeric and categorical features, missingness, and outliers — RF handles all of this without complaint. (c) You need native feature importance for stakeholder explainability — the impurity-based ranking is immediately business-readable. (d) You're doing classification or regression on structured/tabular data and don't need real-time sub-millisecond inference. (e) You want a robust model that won't catastrophically overfit if a junior analyst touches the hyperparameters. PREFER XGBOOST/GBDT WHEN: you need the last 1-2% of accuracy and can invest in careful tuning. PREFER LOGISTIC REGRESSION WHEN: full statistical inference (p-values, confidence intervals) is required. AVOID WHEN: data is sequential/temporal (use LSTM, temporal transformers), image/audio (use CNNs/transformers), or you need real-time serving at extreme scale (a 500-tree forest predicting on millions of records per second requires optimization).

PROBLEM — D&B Churn Risk Scoring for Finance Vertical Accounts: D&B's retention team wants to identify which mid-market Finance sector accounts are at risk of not renewing their data subscription in the next 90 days, so the account expansion agent archetype can intervene proactively. FEATURES: Recency of API calls (days since last hit), monthly query volume trend (3-month slope), number of DUNS records accessed, Paydex score volatility of their monitored portfolio, support ticket frequency, contract tenure, ARR band, number of active integrations (e.g., Salesforce connector active Y/N), days to renewal. TARGET: Binary — churned within 90 days (1) or retained (0). Labels from 18 months of historical renewal data. BUILD: Train a Random Forest (500 trees, max_features='sqrt', class_weight='balanced' to handle churn imbalance ~15%). OOB AUC comes back at 0.81 — strong signal. FEATURE IMPORTANCE RESULT: Top 3 predictors are (1) monthly query volume 3-month slope going negative, (2) days since last API hit > 45, (3) number of active integrations = 0. BUSINESS TRANSLATION: Accounts going quiet on API usage and running no integrations are the canary in the coal mine — they've already mentally churned before they formally cancel. OPERATIONALIZATION: Scores refresh weekly via a batch MLflow pipeline. Accounts crossing a 0.65 probability threshold trigger an automated task in Salesforce Agentforce for the AE, surfacing the specific risk factors (not just the score — this is the 'Explainability First' principle in action). The entity resolution layer ensures that subsidiary-level API activity rolls up correctly to the corporate parent via DUNS corporate family trees, so a Fortune 500 subsidiary going quiet doesn't get missed because the parent account looks healthy.
