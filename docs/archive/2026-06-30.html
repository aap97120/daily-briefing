🧠 Data Science Topic of the Day

🎓 SHAP (SHapley Additive exPlanations)

Make any ML model explain itself — feature by feature, prediction by prediction.

What It Is

SHAP is a method from cooperative game theory that assigns each input feature a fair share of the credit (or blame) for a model's prediction. It answers the question: 'Why did this model score this record this way?' Rather than a single global feature importance list, SHAP produces a local explanation for every individual prediction, and those local explanations are guaranteed to be consistent and additive — they sum exactly to the difference between the prediction and the baseline average. This makes SHAP the gold standard for explainable AI (XAI) in production ML systems, and it works on any model: XGBoost, Random Forests, neural nets, linear models.

How It Works

The core idea comes from Shapley values in game theory. Imagine your model's features are players in a coalition game, and the 'payout' is the prediction. SHAP asks: if we try every possible subset (coalition) of features, how much does adding feature X change the prediction on average? That marginal contribution, averaged across all possible orderings of features entering the coalition, is the Shapley value. In practice, exact computation is exponential in the number of features, so the SHAP library uses model-specific fast approximations: TreeSHAP for tree ensembles (exact, polynomial time), LinearSHAP for linear models, and KernelSHAP as a model-agnostic fallback. The output is a SHAP value per feature per row — positive means the feature pushed the prediction above the baseline, negative means it pushed it below. Key visualizations: (1) Waterfall plot — shows a single prediction decomposed feature by feature. (2) Beeswarm / summary plot — global view of which features matter most across the whole dataset. (3) Dependence plot — how one feature's SHAP value changes as its raw value changes, revealing non-linearities and interactions.

When To Use

Use SHAP when: (1) You need to explain individual decisions to a customer, regulator, or sales rep — not just 'the model said so.' (2) You're debugging a model that's behaving unexpectedly and need to trace which features are misfiring. (3) You want to audit a model for bias — do certain DUNS attributes or industry codes disproportionately drive scores? (4) You're building a feature store and need to validate that newly engineered features are adding signal, not noise. (5) You operate in a compliance environment (credit decisioning, fraud, KYC) where adverse action notices require per-decision explanations. Avoid KernelSHAP on very large datasets — it's slow. Prefer TreeSHAP for any gradient boosted or Random Forest model in production.
