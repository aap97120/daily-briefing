🧠 Data Science Topic of the Day

🎓 Model Context Protocol (MCP)

The universal 'USB-C port' that lets AI agents plug into any data source or tool — including D&B's entire intelligence stack.

What It Is

Model Context Protocol (MCP) is an open standard introduced by Anthropic in late 2024 that defines a common interface between AI language models (or agents) and external tools, APIs, and data sources. Before MCP, every agent integration was bespoke — you wrote custom glue code for each LLM-to-tool connection. MCP standardises that handshake: one server exposes capabilities (tools, resources, prompts), and any MCP-compatible client (Claude, GPT-4o, Gemini, LangChain, etc.) can discover and invoke them without custom adapters. Think of it as the HTTP of agentic AI — a protocol layer, not a product.

How It Works

MCP operates on a client-server model with three primitives: (1) TOOLS — callable functions the model can invoke (e.g. 'look up DUNS number', 'score credit risk'); (2) RESOURCES — structured data the model can read (e.g. a company's corporate family tree, a Paydex score time series); (3) PROMPTS — reusable prompt templates the server exposes to guide the model's behaviour. At runtime: the MCP Client (inside the AI agent) sends a JSON-RPC 2.0 request to the MCP Server listing available tools/resources. The LLM reasons about which tool to call, the client executes it against the server, and the result is injected back into the model's context. Critically, the model never directly touches your database — the MCP Server acts as a governed, auditable proxy. Transport can be local (stdio) for desktop tools or remote (HTTP + Server-Sent Events) for cloud APIs. Because the protocol is stateful, the server can maintain session context — e.g. remembering that 'Acme Corp' was already resolved to DUNS 123456789 earlier in the conversation.

When To Use

Use MCP when: (1) you want one canonical integration that works across multiple AI platforms simultaneously rather than maintaining separate connectors for each; (2) you need governed, auditable tool access — MCP servers can enforce auth, rate limits, and logging at the protocol layer; (3) your agents need to compose multiple data sources dynamically at runtime (e.g. resolve an entity via D&B, then pull financials, then check trade credit — all in one agent turn); (4) you are building agentic workflows inside enterprise platforms (Microsoft Copilot, Salesforce Agentforce, ServiceNow) that are adopting MCP natively. Avoid MCP for simple single-call API wrappers where the overhead of a server is unnecessary, or for latency-critical pipelines where the extra JSON-RPC round-trip matters.

Scenario — D&B 'Smart Prospecting' MCP Server: D&B builds and publishes a single MCP Server that exposes five tools: resolve_entity(name, address) → DUNS + canonical company record; get_firmographics(duns) → industry, employee count, revenue band; get_paydex(duns) → payment behaviour score + trend; get_corporate_family(duns) → parent/subsidiary tree with DUNS IDs; get_intent_signals(duns) → Buydex buying-window score. The agent returns a ranked, explainable prospect list — citing which D&B signal drove each ranking. Crucially, Salesforce Agentforce and Google Gemini for Workspace can point at the SAME MCP server with zero additional integration work. D&B writes the server once; it is embedded everywhere.
